<h2 id="what-is">
<a aria-hidden="true" class="anchor-link" href="#what-is" tabindex="-1"></a>What is JavaScript SEO?</h2>
<p>JavaScript SEO encompasses everything you need to do to make a JavaScript-reliant website perform well in search engines. Think of it as a specialization within Technical SEO.</p>
<h2 id="why-important">
<a aria-hidden="true" class="anchor-link" href="#why-important" tabindex="-1"></a>Why is it important?</h2>
<p>Developers will tell you JavaScript is amazing and rave on about AngularJS, Vue, Backbone, or React. They‚Äôre in love with JavaScript because it lets them create highly interactive web pages that people love.</p>
<p>SEOs will tell you that JavaScript is often horrible for your SEO performance, and they‚Äôll even go as far as saying that JavaScript is job security for them while quickly pulling up charts showing sharp declining organic traffic after a site starts relying on client-side rendering.</p>
<p>Both are right.</p>
<p>At the same time, when developers and SEOs work together productively, they can achieve great results. When they focus on creating the best possible experience for both visitors and search engines, <mark>even JavaScript-reliant websites can perform very well in search</mark>.</p>
<p>For a JavaScript-reliant website to perform well in search, it‚Äôs absolutely vital that search engines be able to fully understand what your pages are about and what your crawling and indexing guidelines are from the initial HTML response.</p>

<h3 id="what-is-javascript">
<a aria-hidden="true" class="anchor-link" href="#what-is-javascript" tabindex="-1"></a>But, what is JavaScript?</h3>
<p>Glad you asked! JavaScript is a programming language for the web that runs locally within your browser. It‚Äôs used to make web pages come to life. For instance, it‚Äôs used to deliver notifications, personalize websites, automatically pull in content updates, or load new content as you almost reach the bottom of a page.</p>
<p>Your most visited websites are heavily relying on JavaScript to function!</p>
<p>Twitter, Facebook, Instagram, YouTube, Netflix‚Äîthey all use JavaScript.</p>
<p>And it‚Äôs <strong>highly unlikely</strong> that your own site doesn‚Äôt use JavaScript. It‚Äôs everywhere, and it‚Äôs not going away. <mark>JavaScript adoption will only grow further</mark>.</p>
<p>In a 2019 survey, <a href="https://fossbytes.com/javascript-most-popular-programming-language-stack-overflow/" rel="noopener" target="_blank"><span class="extlink">67.8% of developers</span><span class="show-for-sr"> (opens in a new tab)</span></a> said they were using JavaScript. <a href="https://octoverse.github.com/" rel="noopener" target="_blank"><span class="extlink">GitHub confirms</span><span class="show-for-sr"> (opens in a new tab)</span></a> its popularity‚Äîand shows that it has consistently been the most popular programming language for several consecutive years now.</p>

<h2 id="impact-seo">
<a aria-hidden="true" class="anchor-link" href="#impact-seo" tabindex="-1"></a>How does JavaScript impact your SEO performance?</h2>
<p>Web pages that heavily rely on JavaScript get indexed slowly. Before we can explain why, we first need to touch on how HTTP requests work and how JavaScript affects web pages.</p>
<h3 id="how-http-requests-work">
<a aria-hidden="true" class="anchor-link" href="#how-http-requests-work" tabindex="-1"></a>How HTTP requests work</h3>
<p>When you navigate to a URL, your browser requests the URL‚Äôs content from the server. If the request was successful, the server responds with the HTML document for that URL. This HTML document contains the text, with references to external files such as images, CSS, and JavaScript if these are present. If they are, your browser creates additional, separate requests for these files too.</p>
<h3 id="javascript-execution-comes-into-play">
<a aria-hidden="true" class="anchor-link" href="#javascript-execution-comes-into-play" tabindex="-1"></a>JavaScript execution comes into play</h3>
<p>The next step is for your browser to construct the DOM and start rendering the web page. Part of this process is executing the JavaScript code which modifies the DOM. These can be small modifications (e.g. loading support chat) or big ones (e.g. loading all of the page‚Äôs content).</p>
<p>The page appears in your browser, and you can interact with it. But, JavaScript rendering that heavily modifies the DOM can add seconds to the time it takes for the page to become interactive for visitors.</p>
<p>We call these modifications to the DOM <em>client-side rendering</em> ("CSR" for short)‚ÄîJavaScript rendering that‚Äôs done on the client side, in this case your browser. JavaScript rendering can be slow for various reasons, such as for example a lot of content that needs to be pulled in, a lot of additional requests that need to be made, and inefficient code.</p>
<p>A lot of WordPress themes, for example, are horribly bloated with JavaScript because they load lots of‚Äîoften unnecessary‚ÄîJavaScript libraries. Google's Lighthouse goes as far as to even <a href="https://searchengineland.com/googles-lighthouse-is-now-recommending-javascript-library-alternates-344940" rel="noopener" target="_blank"><span class="extlink">propose alternative JavaScript libraries</span><span class="show-for-sr"> (opens in a new tab)</span></a>.</p>

<div aria-hidden="true" class="quote-box__avatar"><span class="avatar avatar--is-flipped avatar--small-size"><span class="avatar__img"><picture><source srcset="/media/people/aleyda-solis@1x.avif, /media/people/aleyda-solis@2x.avif 2x" type="image/avif"/><source srcset="/media/people/aleyda-solis@1x.webp, /media/people/aleyda-solis@2x.webp 2x" type="image/webp"/><img alt="Aleyda Solis" class="image" height="100" loading="lazy" src="/media/people/aleyda-solis@1x.jpg" srcset="/media/people/aleyda-solis@1x.jpg, /media/people/aleyda-solis@2x.jpg 2x" title="Aleyda Solis" width="100"/></picture></span></span></div>

<blockquote class="quote-box__blockquote">
<p>Just in the last few months, I've had 3 clients using JavaScript to not only implement but also <em>change</em> some pages meta data, as well as canonical tags and main navigation elements. These were large websites, with millions of URLs that were also suffering from speed issues with non-trivial rendering times (partially caused by the high usage of JavaScript).</p>
<p>Although in some scenarios non-trivial changes needed to be made within the platforms, it was clear that the way JavaScript was being used here was unnecessary and harming the ability of the site to be fully and consistently crawled and indexed.</p>
<p>Ultimately, this lead to a bad page experience due to the higher loading times, and needed to be addressed. When the changes started to be made, it was really noticeable in the websites crawling, indexing and speed performance.</p>

<h3 id="what-is-the-dom">
<a aria-hidden="true" class="anchor-link" href="#what-is-the-dom" tabindex="-1"></a>What is the DOM?</h3>
<p>Above we mentioned the ‚ÄúDOM‚Äù. The DOM is an object-oriented representation of a web page, which can be modified using a programming language such as JavaScript.</p>
<figure aria-labelledby="image-caption-1" class="captioned-image"><div class="media-container" style="max-width: 600px;"><picture><source media="(max-width: 600px)" srcset="/media/content/contentking-javascript-seo-dom-tree@1x.avif" type="image/avif"/><source srcset="/media/content/contentking-javascript-seo-dom-tree@1x.avif, /media/content/contentking-javascript-seo-dom-tree@2x.avif 2x" type="image/avif"/><source media="(max-width: 600px)" srcset="/media/content/contentking-javascript-seo-dom-tree@1x.webp" type="image/webp"/><source srcset="/media/content/contentking-javascript-seo-dom-tree@1x.webp, /media/content/contentking-javascript-seo-dom-tree@2x.webp 2x" type="image/webp"/><source media="(max-width: 600px)" srcset="/media/content/contentking-javascript-seo-dom-tree@1x.png"/><img alt="DOM tree explained" class="image" height="274" loading="lazy" src="/media/content/contentking-javascript-seo-dom-tree@1x.png" srcset="/media/content/contentking-javascript-seo-dom-tree@1x.png, /media/content/contentking-javascript-seo-dom-tree@2x.png 2x" title="DOM tree explained" width="600"/></picture></div>
<figcaption class="captioned-image__caption" id="image-caption-1">Visual representation of a DOM tree</figcaption></figure>
<p>DOM stands for <em>Document Object Model</em>:</p>
<ul>
<li>
<strong>Document</strong>: This is the web page.</li>
<li>
<strong>Object</strong>: Every element on the web page (e.g. <code class="language-plaintext highlighter-rouge">&lt;head&gt;&lt;/head&gt;</code>, <code class="language-plaintext highlighter-rouge">&lt;body&gt;&lt;/body&gt;</code>, <code class="language-plaintext highlighter-rouge">&lt;title&gt;&lt;/title&gt;</code>, <code class="language-plaintext highlighter-rouge">&lt;h1&gt;&lt;/h1&gt;</code> and <code class="language-plaintext highlighter-rouge">&lt;p&gt;&lt;/p&gt;</code>).</li>
<li>
<strong>Model</strong>: Describes the hierarchy within the document (e.g. <code class="language-plaintext highlighter-rouge">&lt;title&gt;&lt;/title&gt;</code> goes into the <code class="language-plaintext highlighter-rouge">&lt;head&gt;&lt;/head&gt;</code>).</li>
</ul>
<h3 id="the-issues-this-presents">
<a aria-hidden="true" class="anchor-link" href="#the-issues-this-presents" tabindex="-1"></a>The issues this presents</h3>
<p>If it took your browser several seconds to fully render the web page, and the page source didn‚Äôt contain much body content, then how will search engines figure out what this page is about?</p>
<p>They‚Äôll need to render the page, similar to what your browser just did, but without having to display it on a screen. Search engines use a so-called ‚Äúheadless browser.‚Äù</p>
<p>Compared to just downloading HTML, rendering JavaScript is super resource-heavy (think 100 times more expensive). That‚Äôs why search engines don‚Äôt render every page they come across.</p>
<p>Back in July 2016, Google said they <a href="https://www.youtube.com/watch?v=PFwUbgvpdaQ&amp;t=245" rel="noopener" target="_blank"><span class="extlink">found over 130 trillion documents</span><span class="show-for-sr"> (opens in a new tab)</span></a>, and it‚Äôs safe to say that since then, that amount of documents has increased massively.</p>
<p>Google simply doesn‚Äôt have the capacity to render all of these pages. They don‚Äôt even have the capacity to crawl all of these pages‚Äîwhich is why every website has an assigned <a href="/academy/crawl-budget/">crawl budget</a>.</p>
<p>Websites have an assigned <a href="/academy/javascript-seo/#render-budget">render budget</a> as well. This lets Google prioritize their rendering efforts, meaning they can dedicate more time to rendering pages that they expect visitors to search for more often.</p>
<p>This in turn causes heavily-reliant JavaScript web pages to get indexed much slower than ‚Äúregular‚Äù HTML pages. And <mark>if your web pages aren‚Äôt indexed, they can‚Äôt rank, and you won‚Äôt get any organic traffic</mark>.</p>

<h2 id="how-google-deals-with-js">
<a aria-hidden="true" class="anchor-link" href="#how-google-deals-with-js" tabindex="-1"></a>How does Google deal with JavaScript sites?</h2>
<figure aria-labelledby="image-caption-2" class="captioned-image"><div class="media-container" style="max-width: 600px;"><a class="media-link" href="https://web.dev/javascript-and-google-search-io-2019/#learn-how-googlebot-works" rel="noopener" target="_blank"><span class="extlink"><picture><source media="(max-width: 600px)" srcset="/media/content/contentking-javascript-seo-googlebot-pipeline@1x.avif" type="image/avif"/><source srcset="/media/content/contentking-javascript-seo-googlebot-pipeline@1x.avif, /media/content/contentking-javascript-seo-googlebot-pipeline@2x.avif 2x" type="image/avif"/><source media="(max-width: 600px)" srcset="/media/content/contentking-javascript-seo-googlebot-pipeline@1x.webp" type="image/webp"/><source srcset="/media/content/contentking-javascript-seo-googlebot-pipeline@1x.webp, /media/content/contentking-javascript-seo-googlebot-pipeline@2x.webp 2x" type="image/webp"/><source media="(max-width: 600px)" srcset="/media/content/contentking-javascript-seo-googlebot-pipeline@1x.png"/><img alt="Google's crawling, indexing, rendering and ranking pipeline" class="image" height="335" loading="lazy" src="/media/content/contentking-javascript-seo-googlebot-pipeline@1x.png" srcset="/media/content/contentking-javascript-seo-googlebot-pipeline@1x.png, /media/content/contentking-javascript-seo-googlebot-pipeline@2x.png 2x" title="Google's crawling, indexing, rendering and ranking pipeline" width="600"/></picture></span><span class="show-for-sr"> (opens in a new tab)</span></a></div>
<figcaption class="captioned-image__caption" id="image-caption-2">Google's pipeline for crawling, indexing, rendering and ranking</figcaption></figure>
<p>The illustration above explains Google‚Äôs processes from crawling to ranking conceptually. It has been greatly simplified; in reality, <a href="https://twitter.com/methode/status/1293229291321397251" rel="noopener" target="_blank"><span class="extlink">thousands of sub-processes</span><span class="show-for-sr"> (opens in a new tab)</span></a> are involved.</p>
<p>We‚Äôll explain every step of the process:</p>
<ol>
<li>
<strong>Crawl Queue:</strong> It keeps track of every URL that needs to be crawled, and it is updated continuously.</li>
<li>
<strong>Crawler</strong>: When the crawler (‚ÄúGooglebot‚Äù) receives URLs from the <code class="language-plaintext highlighter-rouge">Crawl Queue</code>, they request its HTML.</li>
<li>
<strong>Processing</strong>: The HTML is analyzed, and <br/>
 a) URLs found are passed on to the <code class="language-plaintext highlighter-rouge">Crawl Queue</code> for crawling. <br/>
 b) The need for indexing is assessed‚Äîfor instance if the HTML contains a <code class="language-plaintext highlighter-rouge">meta robots noindex</code>, then it won‚Äôt be indexed (and will not be rendered either!). The HTML will also be checked for any new and changed content. If the content didn‚Äôt change, the index isn‚Äôt updated. <br/>
 c) The need for rendering is assessed, based on the page‚Äôs reliance on JavaScript. URLs that need to be rendered are added to the <code class="language-plaintext highlighter-rouge">Render Queue</code>. <mark>Please note that Google can already use the initial HTML response while rendering is still in progress</mark>. <br/>
 d) URLs are canonicalized (note that this goes beyond the canonical link element; other canonicalization signals such as for example the XML sitemaps and internal links are taken into account as well).</li>
<li>
<strong>Render Queue</strong>: It keeps track of every URL that needs to be rendered, and‚Äîsimilar to the <code class="language-plaintext highlighter-rouge">Crawl Queue</code>‚Äîit‚Äôs updated continuously.</li>
<li>
<strong>Renderer</strong>: When the renderer (<em>Web Rendering Services</em>, or ‚ÄúWRS‚Äù for short) receives URLs, it renders them and sends back the rendered HTML for processing. Steps 3a, 3b, and 3d are repeated, but now using the <em>rendered</em> HTML.</li>
<li>
<strong>Index</strong>: It analyzes content to determine relevance, structured data, and links, and it (re)calculates the PageRank and lay-out.</li>
<li>
<strong>Ranking</strong>: The ranking algorithm pulls information from the index to provide Google users with the most relevant results.</li>
</ol>
<p>So, in order for Google to understand a JavaScript-reliant web page, it needs to go through the full rendering phase, instead of being able to directly</p>
<ol>
<li>forward URLs that need to be crawled to the <code class="language-plaintext highlighter-rouge">Crawl Queue</code> and</li>
<li>forward information that needs to be indexed to the <code class="language-plaintext highlighter-rouge">Index</code> phase. This makes the whole crawling and indexing process very inefficient and slow.</li>
</ol>
<p>Imagine having a site with 50,000 pages, where Google needs to do a double-pass and render all of those pages. That doesn‚Äôt go down great, and it negatively impacts your SEO performance‚Äîit will take forever for your content to start driving organic traffic and deliver ROI.</p>
<p>Rest assured, when you continue reading you‚Äôll learn how to tackle this.</p>
<section aria-label="Pro tip" class="highlighted-box typo-block highlighted-box--with-icon highlighted-box--protip-style"><div aria-hidden="true" class="highlighted-box__title">Pro tip</div>
<div class="highlighted-box__content">
<p>Don't block CSS and JavaScript files to try and preserve <em>crawl budget</em>, because this prevents search engines from rendering your pages ‚Äî leading to your pages being poorly understood by search engines and an inevitable poor SEO performance.</p>
</div></section>

<h4 id="tabbed-content">Tabbed content</h4>
<p>Please note that this applies to tabbed content as well. For example on product detail pages, make sure content that‚Äôs hidden behind tabs is included in the initial HTML response and doesn‚Äôt require execution of JavaScript code to pull it in.</p>
<h3 id="all-pages-should-have-unique-urls">
<a aria-hidden="true" class="anchor-link" href="#all-pages-should-have-unique-urls" tabindex="-1"></a>All pages should have unique URLs</h3>
<p>Every page on your site needs to have a unique URL; otherwise Google will have a really hard time exploring your site and figuring out what your pages need to rank for.</p>
<p>Don‚Äôt use fragments in URLs to load new pages, as Google will mostly ignore these. While it may be fine for visitors to check out your ‚ÄúAbout Us‚Äù page on <code class="language-plaintext highlighter-rouge">https://example.com#about-us</code>, search engines will often disregard the fragment, meaning they won‚Äôt learn about that URL.</p>
<section aria-label="Did you know..." class="highlighted-box typo-block highlighted-box--with-icon highlighted-box--protip-style"><div aria-hidden="true" class="highlighted-box__title">Did you know...</div>
<div class="highlighted-box__content">
<p>‚Ä¶that technical SEOs recommend using fragments for faceted navigation on places like eCommerce sites to preserve crawl budget?</p>
</div></section>
<h3 id="include-navigational-elements-in-your-initial-html-response">
<a aria-hidden="true" class="anchor-link" href="#include-navigational-elements-in-your-initial-html-response" tabindex="-1"></a>Include navigational elements in your initial HTML response</h3>
<p>All navigational elements should be present in the HTML response. Including your main navigation is a no-brainer, but don‚Äôt forget about your sidebar and footer, which contain important contextual links.</p>
<p>And especially in eCommerce, this one is important: pagination. While infinite scrolling makes for a cool user experience, it doesn‚Äôt work well for search engines, as they don‚Äôt interact with your page. So they can‚Äôt trigger any events required to load additional content.</p>
<p>Here‚Äôs an example of what you need to avoid, as it requires Google to render the page to find the navigation link:</p>
<section aria-label="Bad example of code" class="code-box code-box--negative"><div class="code-box__code">

</div></section>
<p>Instead, do this:</p>
<section aria-label="Good example of code" class="code-box code-box--positive"><div class="code-box__code">
<figure class="highlight"><pre><code class="language-text" data-lang="text">&lt;a href="https://example.com/shoes/"&gt;</code></pre></figure>
</div></section>

<h3 id="send-clear-unambiguous-indexing-signals">
<a aria-hidden="true" class="anchor-link" href="#send-clear-unambiguous-indexing-signals" tabindex="-1"></a>Send clear, unambiguous indexing signals</h3>
<p>Google urges site owners to send clear and unambiguous crawling and indexing signals. Here we‚Äôll look at several directives and explain why communicating clearly is more important than ever when we‚Äôre dealing with JavaScript SEO.</p>
<h4 id="meta-robots-directives">Meta robots directives</h4>
<p>Overwriting your meta robots directives using JavaScript causes trouble. Here‚Äôs why:</p>
<ul>
<li>If you‚Äôve got a <code class="language-plaintext highlighter-rouge">&lt;meta name="robots" content="noindex, follow" /&gt;</code> included in the initial HTML response that you‚Äôve overwritten with a <code class="language-plaintext highlighter-rouge">index, follow</code> value using JavaScript, then Google will not find out about it because upon seeing the <code class="language-plaintext highlighter-rouge">noindex</code>, they decide not to spend precious rendering resources on it.<br/><br/>
On top of that, even if they were to discover that the <code class="language-plaintext highlighter-rouge">noindex</code> has been changed to <code class="language-plaintext highlighter-rouge">index</code>, Google generally adheres to the most restrictive directives, which is the <code class="language-plaintext highlighter-rouge">noindex</code> in this case.<br/>¬†</li>
<li>But what if you do it the other way around, having <code class="language-plaintext highlighter-rouge">&lt;meta name="robots" content="index, follow" /&gt;</code> and then using JavaScript to change that <code class="language-plaintext highlighter-rouge">index</code> to <code class="language-plaintext highlighter-rouge">noindex</code>?<br/><br/>In that case, Google is likely to just index the page because it‚Äôs allowed according to the initial HTML response. However, only after the page has been rendered, Google finds out about the <code class="language-plaintext highlighter-rouge">noindex</code> and removes the page from its index. For a (brief) period of time, that page which you didn‚Äôt want to be indexed was in fact indexed and possibly even ranking.</li>
</ul>

<h4 id="canonical-links">Canonical links</h4>
<p>Overwriting canonical links equally makes mayhem.</p>
<p><a href="https://twitter.com/JohnMu/status/994713207913639939" rel="noopener" target="_blank"><span class="extlink">In May of 2018</span><span class="show-for-sr"> (opens in a new tab)</span></a>, John Mueller said: ‚Äú<em>We (currently) only process the rel=canonical on the initially fetched, non-rendered version.</em>‚Äù and Martin Splitt <a href="https://www.youtube.com/watch?v=bAE3L1E1Fmk&amp;t=772" rel="noopener" target="_blank"><span class="extlink">said in November 2020</span><span class="show-for-sr"> (opens in a new tab)</span></a> that this ‚Äúundefined behaviour‚Äù leads to guesswork on Google‚Äôs part‚Äîand that‚Äôs really something you should avoid.</p>
<p>Injecting canonical links through JavaScript leads to the situation where Google only finds out about the canonical links after they‚Äôve crawled and rendered your pages. We‚Äôve seen instances where Google starts crawling massive amounts of personalized pages with unique URLs because the canonical link wasn‚Äôt included in the initial HTML response, but injected through JavaScript. This is a waste of crawl budget and should be avoided.</p>
<h4 id="relnofollow-link-attribute-value">rel="nofollow" link attribute value</h4>
<p>The same goes for adding the <code class="language-plaintext highlighter-rouge">rel="nofollow"</code> attribute value to links using JavaScript. Google‚Äôs crawler will just extract these links and add them to the Crawl Queue. They get crawled sooner than Google was able to render the page, resulting in the discovery of the <code class="language-plaintext highlighter-rouge">rel="nofollow"</code>. Again, this is a waste of crawl budget and only leads to confusion.</p>
<p>The other way around, having the <code class="language-plaintext highlighter-rouge">rel="nofollow"</code> link attribute value present and removing it using JavaScript won‚Äôt work‚Äîsimilar to meta robots directives‚Äîbecause Google will adhere to <a href="https://www.reddit.com/r/TechSEO/comments/a5uyv5/question_on_js_manipulation_of_nofollow_code/ebric7p/" rel="noopener" target="_blank"><span class="extlink">the most restrictive directive</span><span class="show-for-sr"> (opens in a new tab)</span></a>.</p>

<h3 id="leverage-javascript-caching-and-use-content-hashes">
<a aria-hidden="true" class="anchor-link" href="#leverage-javascript-caching-and-use-content-hashes" tabindex="-1"></a>Leverage JavaScript caching and use content-hashes</h3>
<p>In order not to affect your website‚Äôs user experience, Google caches JavaScript aggressively. That‚Äôs great when your JavaScript code doesn‚Äôt change too often, but what if it does? Then you need to be able to let Google pull the newest version quickly.</p>
<p>Add a <em>content-hash</em> to your JavaScript file names, including file names such as:</p>
<ul>
<li><code class="language-plaintext highlighter-rouge">main.2a846fa617c3361f.js</code></li>
<li><code class="language-plaintext highlighter-rouge">example.117e1c5c1e1838c3.js</code></li>
<li><code class="language-plaintext highlighter-rouge">runtime.9bdf68f2f1992a3d.js</code></li>
</ul>
<p>When your JavaScript code changes, the hash updates, and Google knows that they need to request it.</p>

<h2 id="rendering-options">
<a aria-hidden="true" class="anchor-link" href="#rendering-options" tabindex="-1"></a>Rendering options</h2>
<p>It‚Äôs clear by now that client-side rendering, entirely relying on the client (a browser or crawler) to execute the JavaScript code, negatively impacts the whole crawling, indexing, and ranking process.</p>
<p>But, besides the JavaScript SEO best practices we just covered in the previous section, there are additional steps you can take to prevent JavaScript from dragging your SEO performance down.</p>
<p>While there are lots of rendering options (e.g. pre-rendering) out there, covering them all is outside of the scope of this article. Therefore, we'll cover the most common rendering options to help provide search engines (and users!) a better experience:</p>
<ol>
<li>Server-side rendering</li>
<li>Dynamic rendering</li>
</ol>
<h3 class="toc" id="server-side-rendering">
<a aria-hidden="true" class="anchor-link" href="#server-side-rendering" tabindex="-1"></a>Server-side rendering</h3>
<p>Server-side rendering is the process of rendering web pages on the server before sending them to the client (browser or crawler), instead of just relying on the client to render them.</p>
<p>This rendering process happens real-time, and visitors and e.g. Googlebot are treated the same. JavaScript code can still be leveraged and is executed after the initial load.</p>
<p><strong>Pros</strong></p>
<ul>
<li>Every element that matters for search engines is readily available in the initial HTML response.</li>
<li>It provides a fast <em>First Contentful Paint</em> ("FCP").</li>
</ul>
<p><strong>Cons</strong></p>
<ul>
<li>
<em>Slow Time to First Byte</em> ("TTFB"), because the server has to render web pages on the fly.</li>
</ul>

<h3 class="toc" id="dynamic-rendering">
<a aria-hidden="true" class="anchor-link" href="#dynamic-rendering" tabindex="-1"></a>Dynamic Rendering</h3>
<p>Dynamic Rendering means that a server responds differently based on who made a request. If it‚Äôs a crawler, the server renders the HTML and sends that back to the client, whereas a visitor needs to rely on client-side rendering.</p>
<p>This rendering option is a workaround and should only be used temporarily. While it sounds like cloaking, Google <a href="https://developers.google.com/search/docs/guides/dynamic-rendering#cloaking" rel="noopener" target="_blank"><span class="extlink">doesn‚Äôt consider dynamic rendering cloaking</span><span class="show-for-sr"> (opens in a new tab)</span></a> as long as the dynamic rendering produces the same content to both request types.</p>
<div class="media-container" style="max-width: 600px;"><picture><source media="(max-width: 600px)" srcset="/media/content/contentking-javascript-seo-google-dynamic-rendering-explained@1x.avif" type="image/avif"/><source srcset="/media/content/contentking-javascript-seo-google-dynamic-rendering-explained@1x.avif" type="image/avif"/><source media="(max-width: 600px)" srcset="/media/content/contentking-javascript-seo-google-dynamic-rendering-explained@1x.webp" type="image/webp"/><source srcset="/media/content/contentking-javascript-seo-google-dynamic-rendering-explained@1x.webp" type="image/webp"/><img alt="Dynamic rendering explained" class="image" height="161" loading="lazy" src="/media/content/contentking-javascript-seo-google-dynamic-rendering-explained@1x.png" title="Dynamic rendering explained" width="600"/></picture></div>
<p><strong>Pros</strong></p>
<ul>
<li>Every element that matters for search engines is readily available in the initial HTML response sent to search engines.</li>
<li>It‚Äôs often easier and faster to implement.</li>
</ul>
<p><strong>Cons</strong></p>
<ul>
<li>It makes debugging issues more complex.</li>
</ul>

<h2 id="render-budget">
<a aria-hidden="true" class="anchor-link" href="#render-budget" tabindex="-1"></a>Render budget</h2>
<p>At <a href="https://www.youtube.com/watch?v=PFwUbgvpdaQ&amp;t=851" rel="noopener" target="_blank"><span class="extlink">Google I/O ‚Äò18, Google‚Äôs Tom Greenaway explained</span><span class="show-for-sr"> (opens in a new tab)</span></a>:</p>
<p>‚Äú<em>The rendering of JavaScript power websites in Google Search is deferred until Googlebot has resources available to process that content.</em>‚Äù</p>
<p>And as you can imagine, Google needs to set priorities in rendering, because not all websites are equal. Therefore, websites have an assigned render budget. This allows Google to dedicate more time to rendering pages that they expect visitors to search for more often.</p>
<p>This causes heavily JavaScript-reliant web pages to get indexed much slower than ‚Äúregular‚Äù HTML pages, especially if such a website falls in the last rendering bracket.</p>

<h2 id="what-about-social-media-crawlers">
<a aria-hidden="true" class="anchor-link" href="#what-about-social-media-crawlers" tabindex="-1"></a>What about social media crawlers?</h2>
<p>Social media crawlers like those from Facebook, Twitter, LinkedIn, and Slack need to have easy access to your HTML as well so that they can generate meaningful snippets.</p>
<p>If they can‚Äôt find a page‚Äôs OpenGraph, Twitter Card markup or‚Äîif those aren‚Äôt available‚Äîyour title and meta description, they won‚Äôt be able to generate a snippet. This means your snippet will look bad, and it‚Äôs likely you won‚Äôt get much traffic from these social media platforms.</p>

<h2 class="toc-hidden" id="js-faqs">
<a aria-hidden="true" class="anchor-link" href="#js-faqs" tabindex="-1"></a>FAQs</h2>
<h3 id="-does-google-crawler-run-javascript">
<a aria-hidden="true" class="anchor-link" href="#-does-google-crawler-run-javascript" tabindex="-1"></a>ü§î Does Google crawler run JavaScript?</h3>
<p>No, Google's crawlers don't run JavaScript. JavaScript rendering happens much further down the indexing pipeline, if Google determines it's required to get a good understanding of a page.</p>
<h3 id="-what-do-google-bots-look-for">
<a aria-hidden="true" class="anchor-link" href="#-what-do-google-bots-look-for" tabindex="-1"></a>ü§ñ What do Google bots look for?</h3>
<p>Google's crawlers continuously look for new content, and when they find it a separate process will process it. That process will first look at the initial HTML response. This response needs to include all of the essential content for search engines to understand what the page is about, and what relations it has to other pages.</p>
<h3 id="-can-facebook-render-javascript">
<a aria-hidden="true" class="anchor-link" href="#-can-facebook-render-javascript" tabindex="-1"></a>üßê Can Facebook render JavaScript?</h3>
<p>No, they can't. Therefore, it's highly recommended to either use server-side rendering or dynamic rendering. Otherwise, Facebook (and other social media platforms) won't be able to generate a good snippet for your URL when it's shared.</p>
