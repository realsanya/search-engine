<h1 class="heading__title">HTTP redirect codes for SEO explained</h1>
<div class="heading__params">
<span aria-hidden="true" class="heading__label">
										Last updated:
									</span>
<span aria-label="The date of last update" class="heading__caption">
										April 14, 2022
									</span>

<div class="post-detail__text typo">
<section aria-label="HTTP redirect codes in short" class="highlighted-box highlighted-box--with-background typo-block"><div aria-hidden="true" class="highlighted-box__title">
<span class="highlighted-box__title-icon"><i class="post-type-icon post-type-icon--reversed-style" style="background-image: url(/assets/images/icons/post-types/redirects--reversed.svg)"></i></span>HTTP redirect codes in short
	</div>
<div class="highlighted-box__content">
<p>The HTTP redirect code, <em>redirect</em> for short, is a way to forward visitors and search engines from one URL to another. Redirects are used when moving content to a new URL, when deleting pages or when changing domain names or merging websites.</p>
<p>Whenever possible avoid using redirects. When you have to use them, make sure to follow these best practices:</p>
<ul>
<li>Avoid chained redirects: one redirect should not forward to another redirect.</li>
<li>Redirect to the preferred version of your website, using the right protocol (http or https), domain name (www or non-www) and path notation (with or without trailing slash).</li>
<li>Use a 302 redirect for inactive campaigns.</li>
<li>Use a 301 redirect for content that's permanently removed.</li>
</ul>
</div>
</section>

<h2 id="what-are">
<a aria-hidden="true" class="anchor-link" href="#what-are" tabindex="-1"></a>What are redirects?</h2>
<p>Redirects are a way to forward visitors and search engines to a different URL than the one they requested. Redirects play an important role within SEO, so it's important to know what types of redirects there are, when to use which ones, and how they measure up against one another.</p>

<h2 id="when-to-use-redirects">
<a aria-hidden="true" class="anchor-link" href="#when-to-use-redirects" tabindex="-1"></a>When to use redirects</h2>
<p>You use redirects when you're moving content around and you want the content to retain its value.</p>
<p>The scope differs; you could just be moving one page or you could be migrating an entire website to a different domain.</p>
<h3 id="url-redirect-when-its-just-a-few-urls">
<a aria-hidden="true" class="anchor-link" href="#url-redirect-when-its-just-a-few-urls" tabindex="-1"></a>URL redirect: when it's just a few URLs</h3>
<p>When just a few URLs are impacted, use URL redirects.</p>
<h4 id="example-summer-sale-campaign">Example: summer sale campaign</h4>
<p>Say you have a temporary summer sale campaign for flip flops. When people request the flip flop category page at <code class="language-plaintext highlighter-rouge">https://www.domain.com/flip-flops/</code>, you want them to be redirected to <code class="language-plaintext highlighter-rouge">https://www.domain.com/flip-flop-summer-sale/</code>, which has a different design and contains the limited offer. <code class="language-plaintext highlighter-rouge">https://www.domain.com/flip-flop-summer-sale/</code> will be removed after 2 weeks, and then <code class="language-plaintext highlighter-rouge">https://www.domain.com/flip-flops/</code> is back in play.</p>
<p>In this case, you want search engines to remember <code class="language-plaintext highlighter-rouge">https://www.domain.com/flip-flops/</code> and to keep it in the index, because the redirect was only temporary in nature. In this case <strong>you'd use a 302 redirect</strong>.</p>
<h4 id="example-deleting-pages-with-seo-value">Example: deleting pages with SEO value</h4>
<p>Say some products on your site are not longer available, and they won't return in the future. You delete those product pages, and you 301 redirect these URLs to the most relevant alternative page. Most of the times that's a related product, but if that's not possible you can redirect to the parent category.</p>
<h3 id="website-redirect-when-its-about-an-entire-website">
<a aria-hidden="true" class="anchor-link" href="#website-redirect-when-its-about-an-entire-website" tabindex="-1"></a>Website redirect: when it's about an entire website</h3>
<p>Domain redirects are used when entire domains are affected.</p>
<h4 id="example-switching-domain-names">Example: switching domain names</h4>
<p>Say you're moving your website over to a new domain name, then you need to redirect both visitors and search engines there. Redirection is done on a web server level, and it's similar to redirecting URLs, the scope of the redirect is just very different â€” it affects the whole website.</p>
<h4 id="example-merging-websites">Example: merging websites</h4>
<p>Another example where you'd use a website redirect is when merging websites; this involves one or more websites that are merged into one. This is common after acquisitions, or when satellite websites cease to exist.</p>

<div aria-hidden="true" class="quote-box__avatar"><span class="avatar avatar--small-size"><span class="avatar__img"><picture><source srcset="/media/people/barry-adams@1x.avif, /media/people/barry-adams@2x.avif 2x" type="image/avif"/><source srcset="/media/people/barry-adams@1x.webp, /media/people/barry-adams@2x.webp 2x" type="image/webp"/><img alt="Barry Adams" class="image" height="100" loading="lazy" src="/media/people/barry-adams@1x.jpg" srcset="/media/people/barry-adams@1x.jpg, /media/people/barry-adams@2x.jpg 2x" title="Barry Adams" width="100"/></picture></span></span></div>

<p>Ideally every internal link on a website points directly to the final destination page. Internal redirects can result in crawl waste and loss of link value, so it's a key priority to minimise redirects on your website as much as possible. You can't always control where external links to your site point to, but all your internal links should be accurate and not result in any form of redirect.</p>

<h2 id="different-types-of-redirects">
<a aria-hidden="true" class="anchor-link" href="#different-types-of-redirects" tabindex="-1"></a>What types of redirects are there?</h2>
<p>Redirects can be divided into <strong>server-side redirects</strong> and <strong>client-side redirects</strong>. Both types of redirects can be implemented in various ways.</p>
<p>Be very careful about your choice for the type of redirect: choosing the wrong redirect for the job could lead to SEO troubles.</p>

<h2 id="server-side-redirects">
<a aria-hidden="true" class="anchor-link" href="#server-side-redirects" tabindex="-1"></a>Server-side redirects</h2>
<p>A server-side redirect is a forwarding method in which the server sends a 3xx HTTP status code when a URL is requested. The server determines what URL visitors and search engines should be sent to.</p>
<h3 id="the-most-common-redirect-status-codes-are">
<a aria-hidden="true" class="anchor-link" href="#the-most-common-redirect-status-codes-are" tabindex="-1"></a>The most common redirect status codes are:</h3>
<table>
<tbody>
<tr>
<td><strong>HTTP Status Code</strong></td>
<td><strong>HTTP version</strong></td>
<td><strong>Temporary/Permanent</strong></td>
</tr>
<tr>
<td><a href="/academy/redirects/#301-moved-permanently">301</a></td>
<td><code class="language-plaintext highlighter-rouge">HTTP/1.0</code></td>
<td>Permanent</td>

<h3 id="301-moved-permanently">
<a aria-hidden="true" class="anchor-link" href="#301-moved-permanently" tabindex="-1"></a>301 redirect: Moved permanently</h3>
<h4 id="what-is-a-301-redirect"><strong>What is a 301 redirect?</strong></h4>
<p>A 301 redirect is a server-side redirect which redirects users from URL A to URL B, while signaling to search engines that URL A's content has been <strong>permanently</strong> <strong>moved</strong> to URL B.</p>
<p>When it comes to redirects, the 301 redirect usually is your best choice. It's worth noting that browsers will often cache 301 redirects because of their permanent nature, so keep that in mind when you're dealing with redirects.</p>
<h4 id="the-role-of-the-301-redirect-in-seo">The role of the 301 redirect in SEO</h4>
<p>The general consensus is that a 301 redirect passes 95-99% of the <a href="/academy/authority/#page-authority">page authority</a> from an old URL to a new URL. While a visitor won't notice the difference between a 301 redirect and a 302 redirect, <strong>for a search engine, these are completely different signals</strong>.</p>
<p>Be careful about using 301 redirects when redirecting isn't permanent. If you intent to remove the 301 redirect shortly after implementing it: be prepared to wait several weeks, if not months, to see the redirected URL back in the search engines' indexes.</p>

<h3 id="302-found-moved-temporarily">
<a aria-hidden="true" class="anchor-link" href="#302-found-moved-temporarily" tabindex="-1"></a>302 redirect: Found (HTTP 1.1) / Moved temporarily (HTTP 1.0)</h3>
<h4 id="what-is-a-302-redirect">What is a 302 redirect?</h4>
<p>A 302 redirect is a server-side redirect that redirects users from <code class="language-plaintext highlighter-rouge">URL A</code> to <code class="language-plaintext highlighter-rouge">URL B</code>, while signaling to search engines that <code class="language-plaintext highlighter-rouge">URL A</code>'s content has been <strong>temporarily</strong> moved to <code class="language-plaintext highlighter-rouge">URL B</code>.</p>
<h4 id="the-role-of-the-302-redirect-in-seo">The role of the 302 redirect in SEO</h4>
<p>By default, a 302 redirect passes no page authority from the old URL to a new URL. 302 redirects are rarely used, since in most cases you want to pass a page's authority, and only a 301 redirect will let you do that.</p>
<p>SEO specialists have conducted research on what happens if a 302 redirect is in place for a long time, let's say longer than 6 months. They found that over time, search engines started to regard the 302 redirect as a 301 redirect, due to its permanent nature. This also means that in this situation the 302 redirect will pass page authority similarly to a 301 redirect.</p>
<p>Situations in which to use 302 redirects:</p>
<ul>
<li>When a URL is temporarily moved to another URL;</li>
<li>When you want to redirect visitors to another page based on their location or language preferences. While this isn't recommended by Google, this is one use case for the 302 redirect;</li>
<li>When you have a desktop and mobile website, and you want to redirect desktop visitors that come to the mobile website, and vice versa.</li>
</ul>

<h3 id="303-see-other">
<a aria-hidden="true" class="anchor-link" href="#303-see-other" tabindex="-1"></a>303 redirect: See Other</h3>
<h4 id="what-is-a-303-redirect">What is a 303 redirect?</h4>
<p>A 303 redirect is a server-side redirect that ensures visitors won't be able to re-submit forms when using the Back button in their browser, because the 303 redirect indicates that the follow-up request to the temporary URL should be made using the <code class="language-plaintext highlighter-rouge">GET</code> HTTP method. 303 redirects should only be used for handling form submissions on websites.</p>
<h4 id="the-role-of-the-303-redirect-in-seo">The role of the 303 redirect in SEO</h4>
<p>The 303 redirect doesn't play a role in SEO, as search engines don't perform <code class="language-plaintext highlighter-rouge">POST</code> requests. Don't use the 303 redirect when you need to redirect URLs after content has moved; it's not what it's intended for.</p>
<h3 id="307-temporarily-redirects">
<a aria-hidden="true" class="anchor-link" href="#307-temporarily-redirects" tabindex="-1"></a>307 redirect: Temporarily Redirect</h3>
<h4 id="what-is-a-307-redirect">What is a 307 redirect?</h4>
<p>A 307 redirect is a server-side redirect that is the HTTP 1.1 equivalent of the 302 redirect. The 307 redirect was brought to life to make sure the HTTP method used to make a request doesn't change when the server responds with a redirect. If <code class="language-plaintext highlighter-rouge">GET</code> HTTP method was used, the <code class="language-plaintext highlighter-rouge">GET</code> is passed on as part of the redirect. If <code class="language-plaintext highlighter-rouge">POST</code> was used, then a <code class="language-plaintext highlighter-rouge">POST</code> is passed on as part of the redirect.</p>
<h4 id="the-role-of-the-307-redirect-in-seo">The role of the 307 redirect in SEO</h4>
<p>Similar to the usage of 302 redirects, a 307 redirect should only be used when content has been temporarily moved to another URL. At the moment, it's still unclear how search engines handle the 307 redirect, so we recommend using the 302 redirect instead to indicate that content has temporarily moved.</p>
<h4 id="307-redirect-used-as-an-internal-redirect-in-case-of-hsts">307 redirect used as an internal redirect in case of HSTS</h4>
<p>HTTP Strict Transport Security, HSTS for short, is a way for web servers to declare that web browsers should interact with it using secure HTTPS connections only, and never via the insecure HTTP protocol. That sounds complicated, but conceptually it's not. Let's take an example:</p>
<p>When your browser knows that a website is using HSTS (because it was told so when previously visiting the website), it will force you to only access the HTTPS version of the website by internally redirecting any HTTP URL to HTTPS whenever you try to visit the HTTP version.</p>
<p>The browser will use a 307 to internally redirect from HTTP to HTTPS. So while the 307 redirect is a server-side redirect, it can also be used as an internal redirect within a browser. In reality, it's not a "real redirect" that search engines will encounter, so don't think you're safe from harm when you see the 307 redirect: it could be an internal redirect. Read more about this on <a href="https://www.troyhunt.com/understanding-http-strict-transport/" rel="noopener" target="_blank"><span class="extlink">Troy Hunt's website</span><span class="show-for-sr"> (opens in a new tab)</span></a>.</p>

<p>Similar to the 307 redirect, the 308 redirect keeps the originally used HTTP method. At the moment, it's still unclear how search engines handle the 308 redirect, so we recommend using the 301 redirect instead to indicate that content has permanently moved.</p>

<h2 id="client-side-redirects">
<a aria-hidden="true" class="anchor-link" href="#client-side-redirects" tabindex="-1"></a>Client-side redirects</h2>
<p>A client-side redirect is a forwarding method in which a visitor's browser handles redirection. Using client-side redirects has several drawbacks, namely:</p>
<ul>
<li>
<strong>SEO:</strong> search engines may not pass authority when client-side redirects are used, because they may not be aware of the redirects yet in the case of JavaScript redirects, or they may not treat it as a redirect because of the refresh time. Keep in mind that, in order for search engines to see a JavaScript redirect, they need to execute it. And their resources for JavaScript execution are limited. That's why it usually takes a while before the redirect is picked up and processed. In case of the meta refresh: if the refresh time is set to "0", search engines will likely treat it as a 301 redirect and pass authority.<br/>
If you want to ensure search engines are able to pick up on redirects quickly, then make sure to use server-side redirects. If content has moved permanently, use a 301 redirect. If it has moved temporarily, use a 302 redirect.</li>
<li>
<strong>Back button:</strong> client-side redirects break the Back button. When trying to use the Back button, visitors will immediately be sent to the URL they were redirected to.</li>
<li>
<strong>Support:</strong> not all browsers support client-side redirects.</li>
</ul>
<p>For all these reasons, we <strong>strongly recommend against using client-side redirects</strong>.</p>
<p>Due to the nature of this article, it is still important to describe the client-side redirects:</p>
<ul>
<li><a href="/academy/redirects/#meta-refresh">Meta refresh redirect</a></li>
<li><a href="/academy/redirects/#javascript-redirects">JavaScript redirect</a></li>

<a aria-hidden="true" class="anchor-link" href="#meta-refresh" tabindex="-1"></a>Meta refresh redirect</h3>

It's used to instruct a browser to refresh a page or load another URL after a certain number of seconds.</p>
<p>Here's an example of what the meta refresh redirect looks like to send a visitor to <code class="language-plaintext highlighter-rouge">https://www.contentkingapp.com/</code> after loading a page:</p>

<h2 id="redirection-best-practices">
<a aria-hidden="true" class="anchor-link" href="#redirection-best-practices" tabindex="-1"></a>What are the best practices for redirection?</h2>
<p>When using redirects, keep the best practices below in mind, to make sure you offer your visitors an optimal user experience and preserve as much page authority as you can.</p>
<h3 id="avoid-redirects-whenever-you-can">
<a aria-hidden="true" class="anchor-link" href="#avoid-redirects-whenever-you-can" tabindex="-1"></a>Avoid redirects whenever you can</h3>
<p>Yes, you read it correctly: avoid using redirects whenever you can. They increase load time and waste <a href="/academy/crawl-budget/">crawl budget</a>.</p>
<p>Please note that it's totally fine to use 301 redirects for configuring your web server so that it <a href="/academy/urls/#consistent">serves URLs according your preferences</a>.</p>

<h3 id="relevant-url">
<a aria-hidden="true" class="anchor-link" href="#relevant-url" tabindex="-1"></a>Redirect to a relevant alternative URL</h3>
<p>When implementing redirects, it's essential that you redirect old URLs to the most relevant alternative URL. This is obviously important from a user point of view, and because of that search engines deem it important as well. If you redirect to URLs that aren't relevant, you're risking search engines not consolidating the value of the old URL with the new one. This results in what's called a <a href="/academy/index-coverage/faq/submitted-soft-404/">soft 404 error</a>.</p>

<div aria-hidden="true" class="quote-box__avatar"><span class="avatar avatar--small-size"><span class="avatar__img"><picture><source srcset="/media/people/cyrus-shepard@1x.avif, /media/people/cyrus-shepard@2x.avif 2x" type="image/avif"/><source srcset="/media/people/cyrus-shepard@1x.webp, /media/people/cyrus-shepard@2x.webp 2x" type="image/webp"/><img alt="Cyrus Shepard" class="image" height="100" loading="lazy" src="/media/people/cyrus-shepard@1x.jpg" srcset="/media/people/cyrus-shepard@1x.jpg, /media/people/cyrus-shepard@2x.jpg 2x" title="Cyrus Shepard" width="100"/></picture></span></span></div>
<div class="quote-box__content typo">
<blockquote class="quote-box__blockquote">
<p>When implementing 301 redirects, it's best to use a target URL that closely resembles the original in both topic and intent. Too often, webmasters lazily redirect pages to their homepage or other off-topic pages, which can result in a soft 404, and Google dropping link equity through the redirect.</p>

<h3 id="avoid-chained-redirect">
<a aria-hidden="true" class="anchor-link" href="#avoid-chained-redirect" tabindex="-1"></a>Avoid chained redirects</h3>
<p>When using redirects, be vigilant against <a href="/academy/redirects/faq/redirect-chains/">chained redirects</a>. A chained redirect is when one URL is requested, a redirect is used to redirect it to another URL and in turn this particular URL is redirected as well. It's important to avoid chained redirects for two reasons:</p>
<ol>
<li>Google usually only follows 5 redirects; after that it aborts to avoid potentially getting stuck, and wasting crawl budget. Please note that they may resume following those redirects, where they left off, at a later stage.</li>
<li>Since not all page authority is passed in a redirect, having a chained redirect with one or more extra hops surely won't improve your amount of page authority passed.</li>
</ol>
<p>Example: let's assume <code class="language-plaintext highlighter-rouge">URL A</code> redirects to <code class="language-plaintext highlighter-rouge">URL B</code>, <code class="language-plaintext highlighter-rouge">URL B</code> to <code class="language-plaintext highlighter-rouge">URL C</code> and <code class="language-plaintext highlighter-rouge">URL C</code> in turn redirects to <code class="language-plaintext highlighter-rouge">URL D</code>. Now, let's say you lose 5% of the authority in a redirect. Then the authority arriving at <code class="language-plaintext highlighter-rouge">URL D</code> is: <code class="language-plaintext highlighter-rouge">0.95</code> x <code class="language-plaintext highlighter-rouge">0.95</code> x <code class="language-plaintext highlighter-rouge">0.95</code> = <code class="language-plaintext highlighter-rouge">0.857</code>.</p>
<figure aria-labelledby="image-caption-3" class="captioned-image"><div class="media-container" style="max-width: 1200px;"><picture><source media="(max-width: 600px)" srcset="/wp-content/uploads/2018/06/ContentKing-Redirect-EN-Redirect-chain@1x.png"/><img alt="Chained redirects" class="image" height="354" loading="lazy" src="/wp-content/uploads/2018/06/ContentKing-Redirect-EN-Redirect-chain@1x.png" srcset="/wp-content/uploads/2018/06/ContentKing-Redirect-EN-Redirect-chain@1x.png, /wp-content/uploads/2018/06/ContentKing-Redirect-EN-Redirect-chain@2x.png 2x" title="Chained redirects" width="1200"/></picture></div>
<figcaption class="captioned-image__caption" id="image-caption-3">Chained redirects</figcaption></figure>
<h3 id="redirect-to-preferred-version-of-your-website-url">
<a aria-hidden="true" class="anchor-link" href="#redirect-to-preferred-version-of-your-website-url" tabindex="-1"></a>Redirect to the preferred version of your website URL</h3>
<p>It's important to choose a preferred version of your website URL and stick with it. Below we describe two important topics you need to think about when linking. Consistently using the preferred version of your website URL prevents unnecessary redirects, which cause loss of page authority.</p>
<h4 id="to-www-or-not-to-www">To www or not to www</h4>
<p>For example: if you prefer to have your website on the <code class="language-plaintext highlighter-rouge">www</code> subdomain, make sure requests for URLs <em>without</em> the <code class="language-plaintext highlighter-rouge">www</code> subdomain are 301 redirected to the version <em>with</em> the <code class="language-plaintext highlighter-rouge">www</code> subdomain. Remember: for search engines, <code class="language-plaintext highlighter-rouge">www.example.com</code> and <code class="language-plaintext highlighter-rouge">example.com</code> are two separate domains.</p>
<h4 id="https-or-not">HTTPS or not</h4>
<p>If you're using HTTPS, make sure the HTTP variants of URLs are 301 redirected to their HTTPS counterparts.</p>
<h4 id="to-slash-or-not-to-slash">To slash or not to slash</h4>
<p>Choose your preference when it comes to trailing slashes as well. For search engines, <code class="language-plaintext highlighter-rouge">www.example.com/page-a</code> and <code class="language-plaintext highlighter-rouge">www.example.com/page-a/</code> are different URLs. When both URLs serve the same page, and no <a href="/academy/canonical/">canonical URL</a> or <code class="language-plaintext highlighter-rouge">meta name="robots" content="noindex"</code> is used, this causes you to have duplicate content.</p>
<p>If you prefer having a trailing slash: make sure requests to the version of the page <em>without</em> the trailing slash are redirected to the version <em>with</em> the trailing slash.</p>
<h4 id="casing">Casing</h4>
<p>When people link out to other sites, they sometimes make mistakes. They could introduce a combination of lowercase and uppercase characters into your URL.</p>
<p>Example: <code class="language-plaintext highlighter-rouge">https://www.example.com/about-Us/</code>. In case people request a URL like that, make sure to 301 redirect it to the lower-case version of the URL: <code class="language-plaintext highlighter-rouge">https://www.example.com/about-us/</code> page, to prevent possible duplicate content and unexpected 404 issues.</p>
<h4 id="getting-links-to-the-preferred-version-of-your-website-url">Getting links to the preferred version of your website URL</h4>
<p>People will link to your website the way they want to. Often they won't adhere to your preferred version for your website URL. They may link to your contact page with the URL <code class="language-plaintext highlighter-rouge">http://yourdomain.com/contact</code> while in reality your URL lives at <code class="language-plaintext highlighter-rouge">http://www.yourdomain.com/contact/</code>. When you've correctly configured your web server, visitors will end up at the right URL, but you'll lose a bit of authority in the redirect.</p>
<p>Monitor your inbound links with a tool such as Majestic or Ahrefs and reach out to the ones linking to the <em>wrong</em> version of your URLs.</p>
<h3 id="use-302-redirect-for-inactive-campaign-urls">
<a aria-hidden="true" class="anchor-link" href="#use-302-redirect-for-inactive-campaign-urls" tabindex="-1"></a>Use a 302 redirect for inactive campaign URLs</h3>
<p>A lot of websites use campaign URLs to support their online and offline campaigns. After these campaigns end, their URLs are often just 301 redirected. That's often incorrect, because these campaigns may return one month later.</p>
<p>When a campaign is simply temporarily inactive and will return in the future: use a 302 redirect to be able to keep the campaign URL in the search engine indexes when you need it again.</p>
<h3 id="use-301-redirect-for-campaign-urls-that-will-never-again-be-used">
<a aria-hidden="true" class="anchor-link" href="#use-301-redirect-for-campaign-urls-that-will-never-again-be-used" tabindex="-1"></a>Use 301 redirect for campaign URLs that will never be used again</h3>
<p>When you're absolutely sure a campaign URL will never be used anymore, it's safe to 301 redirect the URL to the most relevant URL within your website. If there's any doubt about it, use a 302 redirect so that you basically have the URL on standby. The URL will remain in the index, and the URL can be used to rank again in the search engines.</p>
<h3 id="clean-up-redirects">
<a aria-hidden="true" class="anchor-link" href="#clean-up-redirects" tabindex="-1"></a>Clean up your redirects</h3>
<p>Having (hundreds of) thousands of redirects may, depending on the way the redirects are implemented, slow a website down significantly.</p>
<p>On top of that, sometimes old redirects can play a vital role in strange website behavior. An old redirect could be causing inexplicable behavior within your website.</p>
<p>Keep things simple; keep the number of redirects within your website to a minimum. Periodically check which redirects are necessary and which can be cleaned up. Only URLs that have inbound links and/or get serious referral traffic should be redirected. These URLs can be found by analyzing the inbound links to pages on your website using services such as Majestic and Ahrefs combined with referral data from your web analytics service. All URLs that redirect but have no inbound links <em>and</em> referring traffic can be removed.</p>

<h3 id="ip-canonicalization">
<a aria-hidden="true" class="anchor-link" href="#ip-canonicalization" tabindex="-1"></a>IP canonicalization</h3>
<p>IP canonicalization is the process of implementing redirects for an IP address to a domain name. This is done to prevent <a href="/academy/duplicate-content/">duplicate content</a>.</p>
<p>If your web server is set up incorrectly, it may serve your website for both your web server's IP address and your domain name. The strange user experience aside, having your website accessible to search engines through both your IP address and your domain name leads to duplicate content and should be avoided.</p>
<p>There's good news: this problem is not hard to solve. For example, if you're using Apache web server, this can be fixed by including the code below in your .htaccess file.</p>
<section aria-label="A fragment of code" class="code-box"><div class="code-box__code">
<figure class="highlight"><pre><code class="language-text" data-lang="text">RewriteEngine On
RewriteCond %{HTTP_HOST} ^123.123.123.123
RewriteRule (.*) http://www.example.com/$1 [R=301,L]</code></pre></figure>
</div></section>
<p>Note: replace <code class="language-plaintext highlighter-rouge">123.123.123.123</code> with your server IP address, and <code class="language-plaintext highlighter-rouge">example.com</code> with your domain name.</p>
<h2 id="setting-up-redirects">
<a aria-hidden="true" class="anchor-link" href="#setting-up-redirects" tabindex="-1"></a>How do you set up redirects?</h2>
<p>The most common ways to set up redirects are through:</p>
<ol>
<li>Web server configuration</li>
<li>Content Management System (CMS)</li>
</ol>
<p>Please note that setting up URL redirects in a CMS has the benefit that it's much more accessible for people without a development- or system administration background, but it has several drawbacks:</p>
<ol>
<li>When there are problems with the CMS, the redirects might stop working.</li>
<li>These redirects are slower, as the CMS needs to be loaded for every redirect.</li>
<li>Redirects set up within the CMS and on the web server level can co-exist, leading to confusion and mistakes.</li>
</ol>
<p>Therefore it's always recommended to set up URL redirects on your web servers, and only use CMS redirects when there is no other suitable way to redirect URLs.</p>

<h3 id="make-setting-up-redirects-easier-using-regular-expression">
<a aria-hidden="true" class="anchor-link" href="#make-setting-up-redirects-easier-using-regular-expression" tabindex="-1"></a>Redirecting using regular expressions</h3>
<p>Regular expressions (RegEx) are a language that can be used to search within text using patterns.</p>
<p>For example, to search a string of text for digits, the regex <code class="language-plaintext highlighter-rouge">[0-9]</code> can be used. Another example is using a regex to match optional characters. <code class="language-plaintext highlighter-rouge">filename.html?</code> matches both <code class="language-plaintext highlighter-rouge">filename.htm</code> and <code class="language-plaintext highlighter-rouge">filename.html</code>.</p>
<p>Regular expressions are extremely useful for setting up redirects, as it saves you the hassle of setting up redirects for individual URLs. For example, let's say you have the following URL structure:</p>
<ul>
<li><code class="language-plaintext highlighter-rouge">/blog/category-1/</code></li>
<li><code class="language-plaintext highlighter-rouge">/blog/category-2/</code></li>
<li><code class="language-plaintext highlighter-rouge">/blog/category-3/</code></li>
</ul>
<p>Then instead of setting up three different redirect rules, you can use the following regular expression to match all these subdirectories: <code class="language-plaintext highlighter-rouge">/blog/(.*)/</code>.</p>
