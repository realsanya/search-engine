<h1 class="heading__title">The Ultimate Guide to Core Web Vitals</h1>
<div class="heading__params">
<span aria-hidden="true" class="heading__label">
										Last updated:
									</span>
<span aria-label="The date of last update" class="heading__caption">

<div class="post-detail__text typo">
<section aria-label="Core Web Vitals in short" class="highlighted-box highlighted-box--with-background typo-block"><div aria-hidden="true" class="highlighted-box__title">
<span class="highlighted-box__title-icon"><i class="post-type-icon post-type-icon--reversed-style" style="background-image: url(/assets/images/icons/post-types/core-web-vitals--reversed.svg)"></i></span>Core Web Vitals in short
	</div>
<div class="highlighted-box__content">
<p>Core Web Vitals are a way to measure the user experience of a page. There are three Core Web Vitals:</p>
<ol>
<li>Largest Contentful Paint (LCP)</li>
<li>First Input Delay (FID)</li>
<li>Cumulative Layout Shift (CLS)</li>
</ol>
<p>If field data shows pages score "good" for all three Core Web Vitals they pass the Core Web Vitals assessment. As of mid-June 2021, Google has started using Core Web Vitals as a ranking signal. But that's not the end of it, only the beginning. Core Web Vitals will only become more important from here on out.</p>
<p>In this article, we'll explain how to measure, interpret and improve your Core Web Vitals to deliver a better user experience, to ultimately make more money!</p>
</div>
</section>

<p>Core Web Vitals are a hot topic in the SEO community right now. For good reason: improving them makes for a better user experience, and also, they are now a ranking factor for Google Search as part of the new page experience signals. The gradual rollout began as of mid-June 2021 and was completed by September 2nd 2021.</p>
<p><mark>In order to pass the Core Web Vitals assessment, you need to score “good” for all three Core Web Vitals—Largest Contentful Paint (LCP), First Input Delay (FID) and Cumulative Layout Shift (CLS)—based on <strong>field data</strong>.</mark></p>

<p>Throughout this article, we’ll frequently talk about field data and lab data. Because these are often misunderstood, we need to explain the difference right away:</p>
<ol>
<li>
<strong>Field data</strong> is gathered from real users through the Chrome User Experience Report (“CrUX” for short).</li>
<li>
<strong>Lab data</strong> is gathered within a controlled environment without any involvement from real users.</li>
</ol>

<p>Now that we have that out of the way, where do Core Web Vitals fit into Google’s ranking algorithms?</p>
<p>Google used to gauge user experience based on whether a site:</p>
<ol>
<li>is mobile-friendly</li>
<li>offers HTTPS</li>
<li>is free of intrusive interstitials</li>
</ol>
<p>Now beside these three factors, they’ve added a fourth—Core Web Vitals—and together these form a group of signals that Google uses to grade “<a href="/blog/page-experience/">Page Experience</a>”, as this handy illustration shows:</p>

<p>It’s important to note that Core Web Vitals—both field data and lab data metrics—will be a moving target: they’re a proxy for measuring user experience, and because the web and its users are constantly evolving, it’s only natural for Core Web Vitals to evolve too.</p>
<p>Why this guide on Core Web Vitals?</p>
<p>Because there is a lot of misinformation about Core Web Vitals—and the correct information is scattered all over the place. Even Google can’t keep everything in one place. By creating this guide, we aim to provide an evergreen resource with <em>correct information</em> about Core Web Vitals, and where to go if you want to dig deeper.</p>

<h2 id="what-are-they">
<a aria-hidden="true" class="anchor-link" href="#what-are-they" tabindex="-1"></a>What are Core Web Vitals?</h2>
<p>We can already hear you reciting the famous Top Gun quote “I feel the need… the need for speed!” in your head, but the Core Web Vitals metrics are about more than <em>just</em> speed.</p>
<p><mark>Core Web Vitals are a set of user-facing metrics related to speed, responsiveness and visual stability, to help site owners measure user experience on the web</mark>.</p>
<p>The Web Vitals metrics are split into <strong>Core Web Vitals</strong> and <strong>non-Core Web Vitals</strong>.</p>
<p>The Core Web Vitals are:</p>
<ul>
<li><a href="/academy/core-web-vitals/#lcp">Largest Contentful Paint (LCP)</a></li>
<li><a href="/academy/core-web-vitals/#fid">First Input Delay (FID)</a></li>
<li><a href="/academy/core-web-vitals/#cls">Cumulative Layout Shift (CLS)</a></li>
</ul>
<p>And the non-Core Web Vitals are:</p>
<ul>
<li><a href="/academy/core-web-vitals/#tbt">Total Blocking Time (TBT)</a></li>
<li><a href="/academy/core-web-vitals/#fcp">First Contentful Paint (FCP)</a></li>
<li><a href="/academy/core-web-vitals/#si">Speed Index (SI)</a></li>
<li><a href="/academy/core-web-vitals/#tti">Time to Interactive (TTI)</a></li>
</ul>
<p>Each metric measures how “good” one part of the Page Experience is. The illustration below shows how a page loads and where the different metrics come into play:</p>

<h2 id="details">
<a aria-hidden="true" class="anchor-link" href="#details" tabindex="-1"></a>Core Web Vitals in detail</h2>
<p>Without further ado, let’s dive into each of the Web Vitals metrics, starting with the Core Web Vitals!</p>
<h3 class="toc" id="lcp">
<a aria-hidden="true" class="anchor-link" href="#lcp" tabindex="-1"></a>Largest Contentful Paint (LCP)</h3>
<p>Largest Contentful Paint (LCP) is a Core Web Vital that measures the time in seconds from when the page starts loading to when the largest text block or image element is rendered on the screen.</p>
<p>Its aim is to measure when the page’s main contents have finished loading. The lower the LCP, the better. A fast LCP reassures users that a page is useful, because it’s a metric that measures perceived load speed. LCP is available in both <strong>field data</strong> and <strong>lab data</strong>.</p>
<p>In the field, the browser will stop reporting new LCP candidates as soon as the user interacts with the page (via tapping, scrolling, a keypress, switching tabs, or closing the tab). In the lab, it’s not entirely clear when the LCP is finished. We expect this moment to be when the page is nearing Time to Interactive (TTI), and it’s clear which element is the final LCP candidate.</p>
<h4 id="important-considerations">Important considerations</h4>
<p>While a page is loading, the largest text block or image element may change, and the most recent candidate is used when measuring the LCP.</p>
<p>For example, consider when at the start an H1 heading is the largest text block, but later on a larger image is loaded. The larger image is then the leading candidate in measuring the LCP.</p>
<div class="media-container" style="max-width: 600px;"><a class="media-link" href="https://web.dev/lcp/#examples" rel="noopener" target="_blank"><span class="extlink"><picture><source media="(max-width: 600px)" srcset="/media/content/contentking-core-web-vitals-lcp-ticker@1x.avif" type="image/avif"/><source srcset="/media/content/contentking-core-web-vitals-lcp-ticker@1x.avif, /media/content/contentking-core-web-vitals-lcp-ticker@2x.avif 2x" type="image/avif"/><source media="(max-width: 600px)" srcset="/media/content/contentking-core-web-vitals-lcp-ticker@1x.webp" type="image/webp"/><source srcset="/media/content/contentking-core-web-vitals-lcp-ticker@1x.webp, /media/content/contentking-core-web-vitals-lcp-ticker@2x.webp 2x" type="image/webp"/><source media="(max-width: 600px)" srcset="/media/content/contentking-core-web-vitals-lcp-ticker@1x.png"/><img alt="LCP Example" class="image" height="233" loading="lazy" src="/media/content/contentking-core-web-vitals-lcp-ticker@1x.png" srcset="/media/content/contentking-core-web-vitals-lcp-ticker@1x.png, /media/content/contentking-core-web-vitals-lcp-ticker@2x.png 2x" title="LCP Example" width="600"/></picture></span><span class="show-for-sr"> (opens in a new tab)</span></a></div>
<p>Please note that <code class="language-plaintext highlighter-rouge">&lt;svg&gt;</code> elements are currently not considered candidates for the Largest Contentful Paint. This was decided to keep things simple, and may change in the future.</p>
<p>It’s unclear whether <code class="language-plaintext highlighter-rouge">&lt;video&gt;</code> elements are currently considered LCP candidates; we’ve reached out to Google for clarification.</p>

<h3 class="toc" id="fid">
<a aria-hidden="true" class="anchor-link" href="#fid" tabindex="-1"></a>First Input Delay (FID)</h3>
<p>First Input Delay (FID) is a Core Web Vital that measures the time in milliseconds from when a user first interacts with your site (i.e. when they click a link, tap a button, or press a key) to when the browser is able to respond to that interaction.</p>
<p>The FID underlies a user’s first impression of your site's interactivity and responsiveness. Better make it a good impression!</p>
<p>As the description of this metric suggests, it is one that can only be measured in the field, as it relies on user interaction. Therefore, the FID is only available in <strong>field data</strong>. The lower the FID, the better.</p>
<p>For testing within the lab, the <a href="/academy/core-web-vitals/#tbt">Total Blocking Time</a> metric is used, as it closely correlates with First Input Delay.</p>
<h4 id="important-considerations-1">Important considerations</h4>
<p>The FID does not measure the time it takes the browser to process an event (e.g. a tap on a button), nor the time it takes to update the user interface after that.</p>
<p>Interactions such as scrolling and zooming aren’t counted as actions, because they’re continuous in nature and have very different performance constraints because the scrolling action is performed by the GPU instead of the CPU, or the CPU's <a href="https://dev.chromium.org/developers/design-documents/compositor-thread-architecture" rel="noopener" target="_blank"><span class="extlink">compositor thread</span><span class="show-for-sr"> (opens in a new tab)</span></a> instead of main thread.</p>
<h4 id="how-to-interpret-your-fid-score">How to interpret your FID score</h4>
<p>Here’s how to interpret your FID score:</p>
<ul>
<li>
<strong>Good</strong>: <code class="language-plaintext highlighter-rouge">&lt;= 100ms</code>
</li>
<li>
<strong>Needs improvement</strong>: <code class="language-plaintext highlighter-rouge">&gt; 100ms and &lt;= 300ms</code>
</li>
<li>
<strong>Poor</strong>: <code class="language-plaintext highlighter-rouge">&gt; 300ms</code>
</li>
</ul>
<h4 id="what-could-be-causing-a-poor-fid-score">What could be causing a poor FID score</h4>
<p>One common reason for a poor FID score is that <a href="https://developer.mozilla.org/en-US/docs/Glossary/Main_thread" rel="noopener" target="_blank"><span class="extlink">a browser’s main thread</span><span class="show-for-sr"> (opens in a new tab)</span></a> is busy parsing and executing JavaScript code. When the main thread is busy, it can’t yet respond to a user’s interaction.</p>
<h4 id="improving-your-fid-score">Improving your FID score</h4>
<p>If you want to improve your FID score, you need to look closely at what’s keeping the browser from going interactive. Examples of things you can do to improve your FID score are:</p>
<ul>
<li>Reducing JavaScript execution time.</li>
<li>Minimizing work done in the main thread.</li>
<li>Reducing the impact of third-party code.</li>
</ul>
<p>Describing the ins and outs of improving your FID score is beyond the scope of this article. So we recommend checking out web.dev’s resources on <a href="https://web.dev/optimize-fid/" rel="noopener" target="_blank"><span class="extlink">optimizing FID scores</span><span class="show-for-sr"> (opens in a new tab)</span></a>.</p>

<h3 class="toc" id="cls">
<a aria-hidden="true" class="anchor-link" href="#cls" tabindex="-1"></a>Cumulative Layout Shift (CLS)</h3>
<p>Cumulative Layout Shift (CLS) is a Core Web Vital that measures the cumulative score of all <em>unexpected</em> layout shifts within the viewport that occur during a page’s entire lifecycle.</p>
<p>Its aim is to measure a page’s “visual stability,” as that heavily influences the user experience. CLS is available in both <strong>field data</strong> and <strong>lab data</strong>. The lower the CLS score, the better the visual stability.</p>
<p>CLS isn’t measured in seconds like most of the other metrics. It works from the viewport size, relates to elements that move between two frames—called unstable elements—and measures their movement in the viewport. The layout shift score is a product of two components: the “impact fraction” and the “distance fraction”.</p>
<p>The “impact fraction” is the area of the viewport that the unstable element takes up in both frames:</p>

<p>The “distance fraction” is the greatest distance the unstable element moves between both frames, divided by the viewport’s largest dimension (width or height):</p>

<p>You’ll find several examples to build an even better understanding of how CLS is calculated <a href="https://web.dev/cls/#layout-shift-score" rel="noopener" target="_blank"><span class="extlink">here</span><span class="show-for-sr"> (opens in a new tab)</span></a>.</p>
<h4 id="important-considerations-2">Important considerations</h4>
<p>“A page’s entire lifecycle” means that when the page remains open for days, or even weeks, the CLS is measured the entire time. Obviously, this is where CLS <strong>field data</strong> and <strong>lab data</strong> will report differences, because tools only gather <strong>lab data</strong> for a very brief period.</p>
<p>Correctly testing for unexpected layout shifts in test environments can prove tricky, because some functionality may be disabled or work differently there. Some examples: cookie notifications may not be shown, live chat support may be disabled, and personalized content will not be loaded.</p>
<h4 id="how-to-interpret-your-cls-score">How to interpret your CLS score</h4>
<p>Here’s how to interpret your CLS score:</p>
<ul>
<li>
<strong>Good</strong>: <code class="language-plaintext highlighter-rouge">&lt;= 0.1</code>
</li>
<li>
<strong>Needs improvement</strong>: <code class="language-plaintext highlighter-rouge">&gt; 0.1 &lt;= 0.25</code>
</li>
<li>
<strong>Poor</strong>: <code class="language-plaintext highlighter-rouge">&gt; 0.25</code>
</li>
</ul>
<h4 id="what-could-be-causing-a-poor-cls-score">What could be causing a poor CLS score</h4>
<p>Unexpected layout shifts often happen because of images or ads with undefined dimensions, resources that are loaded asynchronously, and situations where new DOM elements are dynamically added to a page, above existing content that has already been loaded. This results in content that has already been loaded being pushed away.</p>

<h3 id="tbt">
<a aria-hidden="true" class="anchor-link" href="#tbt" tabindex="-1"></a>Total Blocking Time (TBT)</h3>
<p>Total Blocking Time (TBT) is a non-Core Web Vital that measures the total time in milliseconds between First Contentful Paint (FCP) and Time To Interactive (TTI) where the main thread is blocked long enough to make it unresponsive to user input.</p>
<p>TBT highly correlates with First Input Delay (FID), and is therefore considered the best alternative when testing in a lab environment where real user interaction isn’t possible. While TBT can be gathered in the field, it’s easily influenced by user interaction and doesn’t make for a reliable metric to measure how long it takes for a page to become responsive to user input. Therefore, TBT is only used in <strong>lab data</strong>.</p>
<p>Any task that takes more than 50ms to execute is considered a long task, and the time on top of the 50ms is considered the “blocking time”. TBT is calculated by taking the sum of the blocking portion of all long tasks. For example, if there are three long tasks:</p>
<ol>
<li>Task A takes 75ms (25ms longer than 50ms)</li>
<li>Task B takes 60ms (10ms longer than 50ms)</li>
<li>Task C takes 85ms (35ms longer than 50ms)</li>
</ol>
<p>The TBT is then: 70ms (25+10+35). The lower the TBT, the better.</p>
<h4 id="how-to-interpret-your-tbt-score">How to interpret your TBT score</h4>
<p>Here’s how to interpret your TBT score:</p>
<ul>
<li>
<strong>Good</strong>: <code class="language-plaintext highlighter-rouge">&lt;= 200ms</code>
</li>
<li>
<strong>Needs improvement</strong>: <code class="language-plaintext highlighter-rouge">&gt; 200ms &lt;= 600ms</code>
</li>
<li>
<strong>Poor</strong>: <code class="language-plaintext highlighter-rouge">&gt; 600ms</code>
</li>
</ul>
<h4 id="causes-for-a-poor-tbt-score-and-how-to-improve-it">Causes for a poor TBT score and how to improve it</h4>

<h3 id="fcp">
<a aria-hidden="true" class="anchor-link" href="#fcp" tabindex="-1"></a>First Contentful Paint (FCP)</h3>
<p>First Contentful Paint (FCP) is a non-Core Web Vital that measures the time from when a page starts loading to when any part of that page’s content is rendered on the screen. Having a fast FCP reassures users that something is happening. Within this context, content means text, images (including background images), <code class="language-plaintext highlighter-rouge">&lt;svg&gt;</code> elements, and non-white <code class="language-plaintext highlighter-rouge">&lt;canvas&gt;</code> elements.</p>
<p>FCP is available in both <strong>field data</strong> and <strong>lab data</strong>, and the lower the FCP, the better.</p>
<h4 id="how-to-interpret-your-fcp-score">How to interpret your FCP score</h4>
<p>Here’s how to interpret your FCP score:</p>
<ul>
<li>
<strong>Good</strong>: <code class="language-plaintext highlighter-rouge">&lt;= 1.8s</code>
</li>
<li>
<strong>Needs improvement</strong>: <code class="language-plaintext highlighter-rouge">&gt; 1.8s &lt;= 3s</code>
</li>
<li>
<strong>Poor</strong>: <code class="language-plaintext highlighter-rouge">&gt; 3s</code>
</li>
</ul>
<h4 id="what-could-be-causing-a-poor-fcp-score">What could be causing a poor FCP score</h4>
<p>Common causes for a poor FCP score are high server response times and render-blocking resources.</p>
<h4 id="improving-your-fcp-score">Improving your FCP score</h4>
<p>You can do a lot of things to improve your FCP score, for example eliminating render-blocking resources, removing unused CSS, minifying CSS, and using a CDN.</p>
<p>The topic of improving your FCP score really deserves an article of its own. Until we write one, we highly recommend you check out web.dev’s resources on <a href="https://web.dev/fcp/#how-to-improve-fcp" rel="noopener" target="_blank"><span class="extlink">optimizing FCP scores</span><span class="show-for-sr"> (opens in a new tab)</span></a>.</p>

<h3 id="si">
<a aria-hidden="true" class="anchor-link" href="#si" tabindex="-1"></a>Speed Index (SI)</h3>
<p>Speed Index (SI) is a non-Core Web Vital that measures how quickly the contents of a page are visibly populated during page load. It’s calculated using frame-by-frame analysis of your page’s load behavior, counting the visual progression between frames captured every 100ms.</p>
<p>SI is available in both <strong>field data</strong> and <strong>lab data</strong>.</p>
<h4 id="how-to-interpret-your-si-score">How to interpret your SI score</h4>
<p>Here’s how to interpret your SI score:</p>
<ul>
<li>
<strong>Good</strong>: <code class="language-plaintext highlighter-rouge">&lt;= 3.4s</code>
</li>
<li>
<strong>Needs improvement</strong>: <code class="language-plaintext highlighter-rouge">&gt; 3.4s &lt;= 5.8s</code>
</li>
<li>
<strong>Poor</strong>: <code class="language-plaintext highlighter-rouge">&gt; 5.8s</code>
</li>
</ul>
<h4 id="what-could-be-causing-a-poor-si-score">What could be causing a poor SI score</h4>
<p>Anything preventing the page from loading quickly will hurt your SI score. Some of the causes that are mentioned for the other metrics, such as for example the main thread being blocked, apply here too.</p>
<h4 id="improving-your-si-score">Improving your SI score</h4>
<p>If you focus on improving overall page load performance, you’ll see your SI score improve too. We recommend checking out web.dev’s resource on this <a href="https://web.dev/speed-index/#how-to-improve-your-overall-performance-score" rel="noopener" target="_blank"><span class="extlink">here</span><span class="show-for-sr"> (opens in a new tab)</span></a>.</p>

<h3 id="tti">
<a aria-hidden="true" class="anchor-link" href="#tti" tabindex="-1"></a>Time to Interactive (TTI)</h3>
<p>Time to Interactive (TTI) is a non-Core Web Vital that measures the time from when the page starts loading to when it's fully interactive.</p>
<p>For it to be fully interactive, it needs to:</p>
<ol>
<li>Display useful content (measured by <a href="/academy/core-web-vitals/#fcp">First Contentful Paint</a>).</li>
<li>Have most visible page elements rendered.</li>
<li>Respond to user interactions within 50 ms.</li>
</ol>
<p>While it’s possible to measure TTI in the field, it’s not recommended, because user interaction can heavily influence your page’s TTI. Therefore, you should only use TTI from a <strong>lab data</strong> environment.</p>
<h4 id="how-to-interpret-your-tti-score">How to interpret your TTI score</h4>
<p>Here’s how to interpret your TTI score:</p>
<ul>
<li>
<strong>Good</strong>: <code class="language-plaintext highlighter-rouge">&lt;= 3.8s</code>
</li>
<li>
<strong>Needs improvement</strong>: <code class="language-plaintext highlighter-rouge">&gt; 3.8s &lt;= 7.3s</code>
</li>
<li>
<strong>Poor</strong>: <code class="language-plaintext highlighter-rouge">&gt; 7.3s</code>
</li>
</ul>
<h4 id="what-could-be-causing-a-poor-tti-score">What could be causing a poor TTI score</h4>
<p>Similarly to the Speed Index metric above, many of the things causing poor scores in the other metrics we described apply to TTI too, because it’s a metric that encompasses those other metrics.</p>

<h2 id="comparing-apples-to-apples">
<a aria-hidden="true" class="anchor-link" href="#comparing-apples-to-apples" tabindex="-1"></a>Comparing apples to apples</h2>
<p>When it comes to Core Web Vitals data, it’s essential that we compare apples to apples. That’s why we need to hammer down the differences between <strong>field data</strong> and <strong>lab data</strong>, as well as mobile data and desktop data.</p>
<h3 id="field-data-vs-lab-data">
<a aria-hidden="true" class="anchor-link" href="#field-data-vs-lab-data" tabindex="-1"></a>Field data vs. lab data</h3>
<p>There are two types of data:</p>
<ol>
<li>
<mark><strong>Field data</strong> is collected from from real users</mark>. Each with their unique device and network connection, through the <a href="https://developers.google.com/web/tools/chrome-user-experience-report" rel="noopener" target="_blank"><span class="extlink">Chrome User Experience Report</span><span class="show-for-sr"> (opens in a new tab)</span></a> (“CrUX report” for short).</li>
<li>
<mark><strong>Lab data</strong> is not collected from real users</mark>. It's collected within a controlled environment with a predefined device and network connection settings.</li>
</ol>
<p>It’s essential that you understand the difference between these two data types. Take a second to let it sink in, as it is perhaps the most commonly misunderstood aspect of Core Web Vitals metrics.</p>
<p>You may be getting great scores in Lighthouse (<strong>lab data</strong>) and be high-fiving yourself, while your actual users are having a poor user experience (<strong>field data</strong>). Alternatively, you can have the same in reverse—great scores based on <strong>field data</strong> and poor <strong>lab data</strong> scores!</p>
<p>Then there’s the “Origin Summary,” which is based on <strong>field data</strong> representing <strong>the aggregate experience of all pages served from your site</strong>. Note that if you have some specific page templates that load notoriously slowly, this will hurt your Origin Summary score.</p>

<h4 id="what-else-you-need-to-know-about-field-data">What else you need to know about field data</h4>
<p>Field data:</p>
<ul>
<li>May not be available for your pages if they aren’t getting enough visitors, meaning not enough CrUX data. This goes for both individual URLs and your Origin Summary.</li>
<li>Is less useful for debugging, because after you make some improvements, you’ll need to wait for new CrUX data to come rolling in. Therefore, we recommend relying on <strong>lab data</strong> for debugging purposes.</li>
<li>Includes data from markets you’re not necessarily catering to, as well. For example, if you’re primarily targeting the United States, but you’re getting a lot of traffic from emerging markets that don’t have the same access to fast internet and hardware, you’re going to see very mixed <strong>field data</strong>, simply because the audience your <strong>field data</strong> is collected from is also very mixed.</li>
<li>May also include data from non-indexable pages such as PPC landing pages <a href="https://www.searchenginejournal.com/core-web-vitals-noindex/395578/" rel="noopener" target="_blank"><span class="extlink">(source)</span><span class="show-for-sr"> (opens in a new tab)</span></a>.</li>
</ul>
<h4 id="what-else-you-need-to-know-about-lab-data">What else you need to know about lab data</h4>
<p>Lab data:</p>
<ul>
<li>Is collected (as of April 2021) by emulating a Moto G4 mobile phone with a fast 3G connection.</li>
<li>Doesn’t include user interaction data, because lab data is simulated. That’s why First Input Delay (FID) isn’t available in lab environments.</li>
<li>Produces reproducible results, because you control the hardware and settings (internet connection and CPU performance).</li>
</ul>

<h3 id="psi">
<a aria-hidden="true" class="anchor-link" href="#psi" tabindex="-1"></a>PageSpeed Insights (PSI)</h3>
<p><a href="https://developers.google.com/speed/pagespeed/insights/" rel="noopener" target="_blank"><span class="extlink">Google’s PageSpeed Insights tool</span><span class="show-for-sr"> (opens in a new tab)</span></a> (“PSI” for short) lets you submit a URL, and it will then do three things:</p>
<ol>
<li>Pull up <strong>field data</strong>
</li>
<li>Collect <strong>lab data</strong> by running <a href="https://developers.google.com/web/tools/lighthouse/" rel="noopener" target="_blank"><span class="extlink">Lighthouse</span><span class="show-for-sr"> (opens in a new tab)</span></a>.</li>
<li>Suggest improvements under “Opportunities” and “Diagnostics.”</li>
</ol>

<h2 id="performance-calculation">
<a aria-hidden="true" class="anchor-link" href="#performance-calculation" tabindex="-1"></a>How is your Lighthouse performance score calculated?</h2>
<p>We’ve covered the Web Vitals scores in great detail, but we have yet to touch on how the overall Lighthouse performance score is calculated. First of all, the Lighthouse performance score is based on <strong>lab data</strong>. It’s a value on a scale from 4 (very bad) to 100 (amazing). It’s a weighted average of the Web Vitals’ metric scores, and the scores aren’t all weighted the same.</p>
<p>The weights for the current version —version 8— and the previous version 6/7 are as follows:</p>

<thead>
<tr>

<td>25%</td>
<td>25%</td>
</tr>

<td>30%</td>
<td>25%</td>
</tr>
<tr>
<td>First Contentful Paint</td>
<td>10%</td>
<td>15%</td>
</tr>

<p>Note: vitals with a * are Core Web Vitals</p>
<p><strong>Important notes:</strong></p>
<ol>
<li>The performance score you’ll find in Lighthouse, and other tools that rely on Lighthouse data, is always based on <strong>lab data</strong>.</li>
<li>In the table above, only two Core Web Vitals metrics are shown. That’s because the third one, First Input Delay, isn’t measurable in a lab environment.</li>
</ol>
<p>The Web Vitals metrics scores are calculated based on their performance relative to real website performance data from the <a href="https://httparchive.org/" rel="noopener" target="_blank"><span class="extlink">HTTP Archive</span><span class="show-for-sr"> (opens in a new tab)</span></a>.</p>
<p>For example: a Largest Contentful Paint (LCP) metric value of <code class="language-plaintext highlighter-rouge">1,220</code> ms is mapped to a metric score of <code class="language-plaintext highlighter-rouge">99</code> based on HTTP Archive data.</p>
<p>Lighthouse then uses the following color-coded ranges to “judge” your score:</p>
<ul>
<li>Red (poor): <code class="language-plaintext highlighter-rouge">0</code> to <code class="language-plaintext highlighter-rouge">49</code>
</li>
<li>Orange (needs improvement): from <code class="language-plaintext highlighter-rouge">50</code> to <code class="language-plaintext highlighter-rouge">89</code>
</li>
<li>Green (good): from <code class="language-plaintext highlighter-rouge">90</code> to <code class="language-plaintext highlighter-rouge">100</code>
</li>
</ul>
<p>We highly recommend playing around with the <a href="https://googlechrome.github.io/lighthouse/scorecalc/" rel="noopener" target="_blank"><span class="extlink">Lighthouse Scoring Calculator</span><span class="show-for-sr"> (opens in a new tab)</span></a>, which immediately updates the score when you change the metrics. To save time configuring the calculator, plug in a URL in PageSpeed Insights and click the <code class="language-plaintext highlighter-rouge">See calculator</code> link beneath the table with the <strong>lab data</strong>:</p>

<h2 id="future">
<a aria-hidden="true" class="anchor-link" href="#future" tabindex="-1"></a>The future of Core Web Vitals</h2>
<p>We’ve covered what the Core Web Vitals are right now, but what does the future hold for us? We’re likely to see a lot of changes over the next few years, as Google will continue to tweak them.</p>
<p>Here’s what we know so far:</p>
<ul>
<li>
<strong>The Core Web Vital set will grow over time</strong>: while Google wants to keep the amount of Core Web Vitals as low as possible, and the Web Vitals as easy to understand and measure as possible, it’s likely the set will grow over time. First Contentful Paint (FCP) is a prime candidate to be added for example; see the <a href="https://www.youtube.com/watch?t=3566&amp;v=a6qC9iZ6bY8" rel="noopener" target="_blank"><span class="extlink">Chrome Dev Summit recording</span><span class="show-for-sr"> (opens in a new tab)</span></a> from Feb 12 2021 and the “<a href="https://www.youtube.com/watch?v=iNfz9tg-wyg&amp;t=203" rel="noopener" target="_blank"><span class="extlink">Exploring the future of Core Web Vitals recording</span><span class="show-for-sr"> (opens in a new tab)</span></a>” from Dec 14 2020.</li>
<li>
<strong>Increased weight of Cumulative Layout Shift (CLS) and tweaks</strong>: the weight of CLS is likely <a href="https://www.youtube.com/watch?t=3566&amp;v=a6qC9iZ6bY8" rel="noopener" target="_blank"><span class="extlink">to be increased</span><span class="show-for-sr"> (opens in a new tab)</span></a>, and they’re looking to <a href="https://www.youtube.com/watch?v=iNfz9tg-wyg&amp;t=328" rel="noopener" target="_blank"><span class="extlink">improve the handling of long-lived pages</span><span class="show-for-sr"> (opens in a new tab)</span></a>, as layout shifts now continue to be added to the CLS score after interaction.</li>
<li>
<strong>Support for measuring animation performance</strong>: user experience goes beyond initial page load, so <a href="https://www.youtube.com/watch?v=H89hKw06iWs&amp;t=7234" rel="noopener" target="_blank"><span class="extlink">Google’s looking into adding</span><span class="show-for-sr"> (opens in a new tab)</span></a> metrics that measure animation performance.</li>
<li>
<strong>First Input Delay (FID) could become stricter</strong>: lowering the FID time to 50-75ms <a href="https://www.youtube.com/watch?v=iNfz9tg-wyg&amp;t=265" rel="noopener" target="_blank"><span class="extlink">could measure the user experience more accurately</span><span class="show-for-sr"> (opens in a new tab)</span></a>.</li>
<li>
<strong>Better support for Single Page Applications (SPAs):</strong> when using SPAs, the app transitions do not have performance metrics because they don’t have unique URLs. For example, First Input Delay (FID) and Largest Contentful Paint (LCP) are only measured upon initial load, but the Cumulative Layout Shift keeps increasing with every interaction. Therefore, <a href="https://www.youtube.com/watch?v=iNfz9tg-wyg&amp;t=348" rel="noopener" target="_blank"><span class="extlink">Google is looking into ways</span><span class="show-for-sr"> (opens in a new tab)</span></a> to better measure Web Vitals in SPAs.</li>
</ul>
<p>We additionally expect that:</p>
<ul>
<li>
<strong>Core Web Vitals are a moving target</strong>: right now, the Moto G4 on a fast 3G connection is leading in determining your page’s Core Web Vitals, but as new, more advanced phones are developed and become available, and faster internet connections become the default, we expect Google will default to those.</li>
<li>
<strong>Websites may start considering blocking traffic from non-target markets</strong>: websites that get a lot of traffic from non-target markets that don’t have access to fast hardware and internet connections could consider preventing users from within these markets from accessing the website, thereby preventing a possible negative effect on their Core Web Vital scores.<br/>
<strong>Disclaimer: we recommend against doing this, as this leads to a poor user experience</strong>.</li>
</ul>

<h2 class="toc-hidden" id="faq-cwv">
<a aria-hidden="true" class="anchor-link" href="#faq-cwv" tabindex="-1"></a>Frequently Asked Questions about Core Web Vitals</h2>
<h3 id="1-how-big-of-an-seo-impact-do-core-web-vitals-have">
<a aria-hidden="true" class="anchor-link" href="#1-how-big-of-an-seo-impact-do-core-web-vitals-have" tabindex="-1"></a>1. How big of an SEO impact do Core Web Vitals have?</h3>
<p>Google has said that Core Web Vitals will likely only have a small impact, but we expect Core Web Vitals to become a more important ranking factor in the future. The reason? Everyone loves sites that provide a good user experience. And the Core Web Vitals metrics aim to measure that user experience. <br/>
<br/>
Additionally, searchers are probably less likely to click back to the SERP if you’re providing a good user experience. This behavior is fed back into Google’s algorithm and will positively affect your page’s SEO performance.</p>
<h3 id="2-how-do-i-pass-the-core-web-vitals-assessment">
<a aria-hidden="true" class="anchor-link" href="#2-how-do-i-pass-the-core-web-vitals-assessment" tabindex="-1"></a>2. How do I pass the Core Web Vitals assessment?</h3>
<p>You’ll pass the Core Web Vitals assessment if you score green (“good”) for all three Core Web Vitals: Largest Contentful Paint, First Input Delay and Cumulative Layout Shift, as based on real user data, called “field data”.</p>
<h3 id="3-why-are-lab-data-and-field-data-scores-so-different">
<a aria-hidden="true" class="anchor-link" href="#3-why-are-lab-data-and-field-data-scores-so-different" tabindex="-1"></a>3. Why are lab data and field data scores so different?</h3>
<p><strong>Lab data</strong> is collected in a simulated environment that aims to reliably run the same tests, using the same settings, whereas <strong>field data</strong> is collected from real user behaviour using different devices, with different internet connections, from different locations.</p>
<h3 id="4-why-is-there-no-field-data-for-my-url-or-origin-summary">
<a aria-hidden="true" class="anchor-link" href="#4-why-is-there-no-field-data-for-my-url-or-origin-summary" tabindex="-1"></a>4. Why is there no field data for my URL or Origin Summary?</h3>
<p>For individual URLs, you may see the “Field Data — The Chrome User Experience Report does not have sufficient real-world speed data for this page” notice, and for Origin Summary, “The Chrome User Experience Report does not have sufficient real-world speed data for this origin.” These mean there is not enough CrUX data available to generate a representative anonymized view of your performance.</p>
<p>Simply put, in order for the CrUX data to become available, you need to get more visitors onto your site! It’s likely you’ll then gain access to the Origin Summary first, because that’s an aggregate across all your pages.</p>

<h3 id="5-do-non-indexable-pages-impact-my-core-web-vitals">
<a aria-hidden="true" class="anchor-link" href="#5-do-non-indexable-pages-impact-my-core-web-vitals" tabindex="-1"></a>5. Do non-indexable pages impact my Core Web Vitals?</h3>
<p>It’s possible that pages that aren’t accessible and/or indexable for search engines <em>do</em> influence your Core Web Vitals scores. When asked in February 2021, Google’s John Mueller said they <em>may</em> have an impact.</p>
